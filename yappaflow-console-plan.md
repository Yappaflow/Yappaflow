# Ops Console — Plan

*(Working title. Final name TBD — candidates: "Ops Console", "Mirza Console", "HQ". Decide in session kickoff.)*

A cross-platform admin app for running Yusuf's projects as a business. Desktop (macOS) + mobile (Android), shared codebase. Not a tool customers see — this is for the operator, to watch services, AI spend, funnel health, **and every dollar in and out** across every project in one place.

Yappaflow is the first project plugged in. The shell is multi-project from day one so other projects can slot in later without a rewrite.

## Platforms

- **macOS desktop** — primary daily driver.
- **Android** — companion app for checking on the go and quick finance entries.
- **iOS** — free with the same Tauri v2 mobile codebase. Ship when convenient.
- **Windows / Linux desktop** — not a priority. Will likely work out of the box from the Tauri build, but not tested.

One codebase (Tauri v2 mobile + desktop), one UI, responsive layout. The mobile build is not a stripped-down version of the desktop — it's the same app reshaped for a small screen.

## Architecture — client/server, not local-only

Because the app has to run on two devices and stay in sync, the data model is:

- **Backend service** — small API + Postgres, hosted on Railway (same provider as Yappaflow). Holds the canonical store for Ops Console's own data: finance entries, scheduled entry rules, project configs, cached vendor spend, alerts history. Runs the scheduled pollers that pull data from vendor APIs, and the recurring-entry engine. One user, one API key — no multi-tenant auth needed because this is personal software.
- **Clients (desktop + mobile)** — talk to the backend over HTTPS. Local SQLite cache so the app opens instantly and works offline for read-only views. Write actions queue locally and sync when online.
- **Separation from Yappaflow's own DB** — Ops Console gets its own Postgres (separate DB, can share the Railway project). Ops Console reads *from* Yappaflow's DB (for agency / project counts, payments booked there) but never writes to it.

## Sync — first-class feature

Desktop and mobile are two clients of one truth. Sync is explicit, not an accident of client/server.

### What "synced" means from the user's POV
- A finance entry logged on the phone shows up on the desktop within seconds (whenever desktop checks next).
- A finance entry logged on the desktop shows up on the phone when the phone opens next.
- Entries logged while offline land on the server the moment connectivity returns, in the correct order, with no duplicates.
- Every screen shows the last sync timestamp and a badge for pending local writes. "Sync now" button on every screen.
- Conflicts are resolved as last-write-wins by `updated_at`. In practice, conflicts are nearly impossible for a single user on two devices — but the code handles them correctly anyway.

### How it works technically

**Every row, every table:** `id` (ULID, generated by the client), `created_at`, `updated_at`, `deleted_at` (soft-delete). Client-generated IDs mean idempotent inserts — the server upserts by ID, so re-sending the same write twice never creates duplicates.

**Pull (server → client):** `GET /sync?since=<timestamp>` returns every create/update/delete across all tables since that timestamp. The client applies them to its local SQLite and advances its `last_sync_at`. Called on app open, when the window regains focus, on manual "Sync now", and on a 30-second poll while the app is foregrounded.

**Push (client → server):** Writes go to local SQLite immediately (optimistic UI). A `outbox` table queues the pending HTTP calls. A tiny worker drains the outbox: POST to backend, on success mark as synced, on failure back off and retry. Everything is idempotent so retries are safe.

**Real-time nudge (optional, Phase 2+):** Server-Sent Events from the backend so when the poller creates entries or the other device pushes writes, connected clients get a "something changed, pull now" ping instead of waiting for the next 30-second poll. SSE because it's one-way server→client and simpler than WebSocket. If Phase 1 skips this and only polls, that is fine.

**Backup of sync plumbing:** a sync log table on the backend records every delta served per client. Useful for debugging "why did this entry disappear?" in the first month.

### Why not a sync library like PowerSync / ElectricSQL
Real options, but overkill for one user on two devices. The protocol above is ~200 lines of code on each side and stays under our control. If sync gets painful later (multi-user, richer conflict semantics), PowerSync is the escape hatch — it sits on top of Postgres without a DB change.

## Why multi-project from day one

Yusuf runs several projects. Money, infra, and AI spend cut across all of them. A single-project console means re-doing the shell the first time a second project gets added. Building the project-aware shape now costs a few extra hours and saves a rewrite. For MVP only Yappaflow is wired to live data — the other slots exist as placeholders.

## Why now, Yappaflow-specifically

With the builder-first pivot, Yappaflow has more moving pieces that quietly fail: the MCP service on its own Railway deploy, the 3-stage DeepSeek/OpenRouter split, schema-version drift across `DesignDna` consumers, and adapter skeletons diverging from canonical structure. A console that answers "is everything green, and am I making money this month?" in one glance is high-leverage for a solo operator.

## App shape

### Main Dashboard (opening screen — All Projects view)

The first thing Yusuf sees every time he opens the app. A one-glance summary across every project:

- Big numbers at top: MTD income, MTD outcome, MTD net, current runway.
- Per-project tiles: each project gets a row/card with its health pill (green/yellow/red), MTD spend, any active alerts.
- Recent alerts feed across all projects.
- Upcoming scheduled finance entries in the next 7 days (rent due, subscriptions renewing, etc.).

This view is where the app proves its value in 5 seconds. No scrolling needed, nothing to click to get the headline.

### Navigation

**Desktop:** left sidebar with "All Projects" at the top (= Main Dashboard), then one row per project, then always-visible top-level links to Finance and Alerts.

**Mobile:** bottom tab bar — Home (Main Dashboard), Finance, Alerts, Projects (which lists projects and drills into a per-project detail view).

### Top-level views (cross-project)
- **Main Dashboard** — described above.
- **Finance** — full finance panel with filters, entry forms, scheduled rules, import/export.
- **Alerts** — rolled-up anomalies across all projects and services.

### Per-project views (when a project is selected)
- **Services** — uptime, latency, deploys for that project's infra.
- **AI Vendor** — AI spend and errors for that project (if it uses AI).
- **Business Snapshot** — project-specific funnel metrics.

## Finance — how income and outcome capture actually works

Three ingestion modes, all of which feed the same underlying `FinanceEntry` table:

### Automatic — API-driven
The backend runs scheduled pollers that pull real data from each integrated source and write entries. Deduplication by `(source, external_id)` so re-runs don't double-count.

**Income sources (auto):**
- Yappaflow's main Postgres — any customer payment booked there becomes an income entry tagged `project: yappaflow`.
- Payment processor (Iyzico / Stripe — TBD) — webhook or polling.

**Outcome sources (auto):**
- DeepSeek billing API.
- OpenRouter `/credits` + `/generation` logs.
- Railway billing API.
- Vercel billing API (if used).
- Any third-party SaaS with a billing API we already pay.

Each source has a small adapter on the backend: `fetch → normalize → upsert`. Cadence: hourly for AI vendors (spend is volatile), daily for billing APIs that only update once per day.

### Scheduled — recurring rules
For predictable costs the user sets up once, then forgets:

- Rent: "15,000 TRY, 1st of every month, category: rent, project: none (personal)"
- Domain renewals: "$12 yearly on specific date"
- Any subscription the user wants to track but that has no API

Backend cron runs once a day at 00:05 local time and creates the entries for rules whose scheduled date has arrived. Each rule shows in a "Scheduled entries" UI where Yusuf can edit / pause / delete it. Upcoming scheduled entries also show on the Main Dashboard so nothing is a surprise.

### Manual — one-off entries
For anything the other two modes miss. Normal form: date, amount, currency, category, project tag, note, receipt attachment (optional). Especially important on mobile — "I just paid someone in cash" should be 3 taps.

### All three modes write to the same `FinanceEntry` table
Each entry carries a `source` field (`"manual" | "scheduled:<rule_id>" | "auto:<adapter>"`). Views don't care about the source; they just aggregate. Auto entries can be manually edited if needed (e.g., re-categorize) — the edit is tracked so the next poll doesn't overwrite it.

### Finance views
- P&L: month-to-date net, last 3 months trend, burn rate, runway.
- By category: where the money goes.
- By project: which projects pay for themselves.
- Upcoming: scheduled entries in the next 30 days.
- History: searchable / filterable list of all entries.

## MVP panels beyond Finance

### Services (per-project)
Live status of the selected project's services. For Yappaflow: main API and `apps/yappaflow-mcp` on Railway. Uptime, p50/p95 latency, 5xx rate, last deploy SHA + timestamp. Green / yellow / red pill per service.

### AI Vendor (per-project)
For Yappaflow: DeepSeek + OpenRouter. Today's spend, this month's spend vs budget, error rate, tokens in/out. Broken down by the three stages (analysis / planning / generation). Fallback events timeline. Other projects: blank state until wired.

### Business Snapshot (per-project)
For Yappaflow: projects created, ZIPs exported, active agencies, last 30 days. Top 5 agencies by projects this month. Read-only pull from Yappaflow's Postgres.

### Alerts (cross-project)
Rolled-up anomalies from the last 24h across all panels, all projects. Build breaks, schema drift, latency spikes, budget overruns, unexpected spend, failed payments. Clickable into the underlying panel. Push notifications on mobile for red-level alerts.

## Out of scope for MVP

No write-backs to external systems except Ops Console's own backend. No restarting services, no changing configs, no sending mails, no moving money. Read-only cockpit plus manual/scheduled/auto finance entries. Write actions to external systems earn their place later.

## Stack

### Clients (desktop + mobile)
- **Tauri v2** with mobile support enabled — native shell on macOS / Android / iOS / (incidentally) Windows / Linux. Same Rust + React codebase.
- **React + Vite + TypeScript** for the UI.
- **Responsive layout from day one** — Tailwind-style breakpoints; mobile = bottom tab bar, desktop = sidebar.
- **`@yappaflow/ui`** for tokens and primitives on the shared pieces. Respect the tsup cross-layer + "use client" quirks.
- **TanStack Query** for fetching from the Ops Console backend.
- **Recharts** for time-series.
- **SQLite** via `tauri-plugin-sql` — local cache + offline write queue.
- **Keychain-backed secure store** (platform-specific via Tauri) for the backend API key.
- **Biometric lock on open** (Face ID / Touch ID on macOS, fingerprint on Android) — finance data is sensitive on a device that can be lost.

### Backend
- **Node + Hono** (or Fastify — pick in kickoff) on Railway.
- **Postgres** on Railway (separate DB from Yappaflow's main).
- **Drizzle ORM** — TypeScript types shared with the client.
- **Vendor adapters** live as small modules with one entry point each: `poll(): Promise<FinanceEntry[]>`.
- **Scheduled engine** — simple daily cron + rule table. No need for BullMQ at this scale.
- **Single API key auth** — one user, one key, stored in each client's keychain.

### Theme
Light default, dark toggle mandatory from day one. Both platforms.

## Phases

**Phase 1 — end-to-end skeleton + sync plumbing, one session.** Scaffold backend (Hono + Postgres + one migration with ULID IDs and `created_at` / `updated_at` / `deleted_at` on every table). Implement the `GET /sync?since=` endpoint and the outbox pattern on the client. Scaffold Tauri v2 client (desktop target first, mobile build config in place but not yet tested). Wire `@yappaflow/ui`, theme toggle, responsive shell, sync-status indicator ("Last synced X ago" + pending-writes badge + "Sync now" button). Build Main Dashboard + Finance panel end-to-end on real data with full sync: manual entry + scheduled rule CRUD + one auto adapter (pick the simplest: DeepSeek or OpenRouter) both writing to Postgres via the backend. Optimistic local writes, offline queue drains when online. Services / AI Vendor / Business Snapshot / Alerts panels mocked but responsive.
Deliverable: an app that already earns its space — Finance actually works on desktop, syncs cleanly, survives an offline→online transition without duplicates.

**Phase 2 — mobile build + live ops data + SSE push.** Bring up Android build, verify responsive layout + sync work on phone. Wire remaining vendor adapters (Railway, Vercel, etc.). Wire Services panel (Railway GraphQL) and AI Vendor panel (with the stage breakdown, assuming step zero is done). Add SSE push so desktop and mobile react to changes in near-real-time instead of waiting for the 30-second poll. SQLite cache tuned.
Deliverable: working Android app + live data on desktop + edits on one device appear on the other within seconds.

**Phase 3 — Business Snapshot + Alerts rules + push notifications.** Wire Yappaflow Postgres (RO). Implement anomaly rules (budget overrun, 5xx spike, fallback-rate spike, schema-drift grep, unexpected large charges, failed payments). Push notifications on mobile for red alerts.

**Phase 4 — routines tab.** Fold the scheduled Yappaflow routines we're discussing separately into a dedicated tab: run on demand, view last run, edit thresholds.

**Phase 5 — plug in second project.** Validate the multi-project shell holds up.

**Step zero (before Phase 2's AI Vendor panel can ship):** confirm `apps/yappaflow-mcp` emits `{stage, provider, latency, tokens, cost}` per call. If not, add it.

## Constraints carried forward

- Light default, dark toggle mandatory, both platforms.
- `DesignDna.schemaVersion` is load-bearing elsewhere — if Ops Console ever reads it, bump + consumer-audit rules apply.
